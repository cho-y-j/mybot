# 작업 일지 — 2026-04-08

## 핵심 성과: 전략 분석 패러다임 전환

오늘은 ElectionPulse를 **단순 모니터링 도구**에서 **전략 분석 프로그램**으로 전환한 날.
사용자가 SSW(서승우 청주시장 후보) 캠프로 실사용 테스트하면서 발견한 큰 결함들을 모두 근본 수정.

## 1. 자동 부트스트랩 (가입 즉시 모든 페이지 채워짐)

### 문제
신규 캠프 가입 후 대시보드/리포트/과거 선거 분석 페이지가 전부 비어 있었음. 사용자가 페이지마다 수동으로 트리거해야 했음.

### 해결
- **`app/elections/bootstrap.py` 신설** — `bootstrap_campaign(db, tid, eid)`:
  1. `analyze_history_deep` → `analysis_cache` 저장
  2. `analyze_survey_deep` → 여론조사 분석
  3. `generate_ai_report` → 첫 AI 리포트 (`reports` 테이블)
- **`onboarding.py` apply hook** — 가입 직후 백그라운드 task로 (1) 데이터 수집 → (2) bootstrap_campaign 자동 실행
- **`history_analyzer.py` fallback** — 시장/군수/구청장 raw 데이터 부족 시 자동으로 광역단체장(governor) 데이터로 대체 분석 + `fallback_notice` 메시지
- **Frontend 배너** — fallback_notice를 `dashboard/history` 페이지 상단에 amber 배너로 정직하게 안내

### 메모리
- `feedback_auto_optimize.md` — "신규 캠프는 가입 직후 모든 페이지가 자동으로 채워져야 함"

## 2. 수집기 교육감 하드코딩 제거 (모든 선거 유형 자동 지원)

### 문제
`instant.py`의 `collect_news_now / community_now / youtube_now` 3개 함수가 모두 "교육감"을 하드코딩.
서승우(시장)인데 "서승우 교육감", "서승우 청주 교육감"으로 검색 → 4월 청주시장 경선 기사 안 나오고 1~2월 잡다한 기사만 들어옴.

### 해결
- `election_type` → `ELECTION_ISSUES[election_type].label` 동적 매핑
- 라벨 괄호 설명 제거 ("시장 (기초단체장)" → "시장")
- `require_any`에 일반 선거 단어 + type_label + 지역(시도/short/시군구) 사용
- 검색 키워드 보강: `f"{이름} {type_label}"`, `f"{이름} {region_short} {type_label}"`

### 결과
**Before:** 서승우 45건, 최신 published_at = 2026-01-19
**After:** 서승우 **127건**, 최신 published_at = **2026-04-08 07:32 (오늘)**

## 3. 동명이인 차단 (산은캐피탈 사외이사 등)

### 문제
서승우 후보 데이터에 "[더벨][이사회 분석 | 산은캐피탈] 사외이사 3인 체제" 같은 무관 기사가 들어감. AI 감성 분석이 동명이인 검증 안 함.

### 해결
- **`naver.py` `_apply_homonym_filter`** — `require_name` 옵션 추가 (후보 이름이 본문에 반드시 있어야 통과)
- **`instant.py` `GLOBAL_HOMONYM_BLOCK`** — 모든 후보 공통 차단 사전:
  - 금융권: 산은캐피탈, 캐피탈, 사외이사, 은행장, 증권, 저축은행
  - 군/경찰: 병장, 일병, 이병, 상병, 해병대
  - 스포츠: 야구, 축구, 프로골프, 선수
  - 연예: 배우, 탤런트, 가수, 아이돌, 드라마
  - 학계/일반: 교수님, 박사학위, 논문, 수상자, 장학금
- **AI 검증 단계** — `media_analyzer`에 `is_relevant` 필드 추가, false면 candidate_id NULL 처리 → 화면 자동 제외

### 결과
산은캐피탈 동명이인: **1건 → 0건**

## 4. 전략 4사분면 (가장 큰 패러다임 전환)

### 문제 (사용자 폭발 포인트)
> "미디어 보면 이범석이가 로그인한 본인인듯 나와있어! 경쟁자 위험은 = 나의 강점 이지!!! 우리의 위기 = 입장발표인데 서승우 우리 후보의 위기가 되는게 말이됨!!!"

단순 감성 분석으론 부족. 모든 콘텐츠는 "우리 후보 vs 경쟁자" × "긍정 vs 부정" 4사분면으로 분류돼야 함.

### 4사분면 정의
| | 부정 톤 | 긍정 톤 |
|---|---|---|
| 우리 후보 | 🚨 **위기 (defend)** — 즉시 방어/해명 | ✨ **강점 (promote)** — 확산/콘텐츠화 |
| 경쟁 후보 | 🔥 **공격 기회 (attack)** — 전략적 활용 | ⚠️ **경쟁 위협 (monitor)** — 견제 |

### 해결
**1) DB 스키마 (3 테이블)**
- `strategic_value` (strength/weakness/opportunity/threat/neutral)
- `action_type` (defend/attack/promote/monitor/ignore)
- `action_priority` (high/medium/low)
- `action_summary` (한 줄 액션)
- `is_about_our_candidate`

**2) AI Batch 분석 (`media_analyzer.py` 전면 재작성)**
- `analyze_batch_strategic` — 8건씩 묶어 한 번의 Claude 호출 → 5배 효율
- 캠프별 our_candidate/rivals 동적 인식 (시장/도지사/교육감 무관)

**3) 사건(event) vs 행동(action) 구분 룰** (사용자 핵심 지적)
프롬프트에 명시:
- 우리 후보가 **주체**가 되어 발표/해명/항의/요구 → **strength** (사건이 부정적이어도)
- 우리 후보가 **객체**가 되어 비판/논란/의혹 보도됨 → **weakness**
- 경쟁자가 객체로 비판당함 → **opportunity** (우리 호재)
- 경쟁자가 주체로 발표/성과 → **threat**

**4) Backend API**
- `GET /api/strategy/{election_id}/quadrant` — 4사분면별 통계 + top items + 액션 추천
- `POST /api/strategy/{election_id}/analyze` — 백그라운드 분석 트리거

**5) Frontend `StrategicQuadrant.tsx`**
- Dashboard 메인 최상단에 4카드 그리드
- 색상별 (위기=red, 기회=orange, 강점=blue, 위협=amber)
- 우리 후보 vs 경쟁자 배지 명확화 ("내 캠프" 파란 / "경쟁" 회색)
- 액션 priority 배지 + AI 액션 한 줄 + 원본 링크

### SSW 첫 분석 결과 (70건, 4분 30초)
| 사분면 | 건수 | 예시 |
|---|---|---|
| ✨ 강점 | 11 | "수도권 쓰레기는 수도권에서" → 카드뉴스 확산 |
| 🚨 위기 | 2 | (재분석 후 0 예상 — 입장 발표 strength로 재분류) |
| 🔥 공격 기회 | 17 | "또 뒤집힌 경선판 아수라장" → 불안정성 공격 |
| ⚠️ 위협 | 28 | 청주예술제 45만 동원 → 성과 약화 필요 |

### 메모리
- `feedback_strategic_quadrant.md` — 4사분면 분류 + 액션 추천 룰
- `feedback_event_vs_action.md` — 사건 vs 행동 구분 (능동 행동은 strength)

## 5. 정렬 룰 명확화 (사용자 두 번째 폭발 포인트)

### 문제

> "전체 뉴스는 우리 후보자, 경쟁자까지 모두 나와야 하고, 뉴스 올라간 날짜별로 최신순으로 나와야 한다고. 모든 범용에도 그렇게 적용해야 한다고."

### 해결
- **Backend `/collectors/{id}/news`** — `is_our_candidate.desc()` 제거. `published_at desc, collected_at desc` 만.
- **Frontend `news/page.tsx`** — 두 곳의 sort 모두 우리 후보 우선 비교 제거. 단순 published_at 역순.

### 메모리
- `feedback_list_sort_rule.md` — "시간 기반 목록은 published_at 역순 ONLY. 우리 후보 우선 가중치 절대 금지. 후보별 보기는 필터 UI로."
